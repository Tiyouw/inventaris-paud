# Admin mode, zone isolation, and single-zone report display (v2)

This revision addresses all five issues raised in the v1 review: the admin credential is now configurable via `ADMIN_ACCESS_CODE` env var, slug uniqueness checks are school-scoped, admin mode is read-only (write buttons hidden), zone edit/delete buttons are hidden for admin, and condition logs are always filtered by active item IDs.

Watch for: The `ADMIN_ACCESS_CODE` fallback to `"admin"` applies in all environments, not just development, so a production deploy without the env var set is openly accessible (confirmed). No server-side guard prevents admin writes if the API is called directly; the read-only posture is purely UI-enforced (confirmed).

## High-level view

The admin credential fix replaces the hardcoded `"admin"` with `process.env.ADMIN_ACCESS_CODE || "admin"`. The `|| "admin"` fallback means a forgotten env var in production still leaves the door open — the `.env.example` comment implies this is dev-only, but the runtime code makes no such distinction.

Zone isolation is correctly scoped end-to-end. `zoneSlugExists` filters by `school_id` (or `IS NULL` for ownerless zones), uses `.limit(1).maybeSingle()` to avoid the multi-row throw, and the creation path stores `school_id` on insert.

Admin read-only enforcement hides all create/edit/delete buttons when `isAdmin` is true. The mutation handlers still exist and would execute if called, but with no UI path to trigger them, the practical risk is limited to direct API calls.

The condition logs query now always filters with `.in("item_id", [...activeItemIds])` regardless of whether `schoolId` is present, bounding transfer size proportionally to loaded items.

## Admin credential fallback doesn't distinguish environments

The verify route reads:

```typescript
const adminAccessCode = process.env.ADMIN_ACCESS_CODE || "admin";
```

The `.env.example` documents this as "Falls back to 'admin' in development if not set," but the code path doesn't check `NODE_ENV`. In any deployment where the variable is omitted — including production — the credential is the literal string "admin". The app silently falls through to the default with no warning.

A safer pattern:

```typescript
const adminAccessCode = process.env.ADMIN_ACCESS_CODE
  || (process.env.NODE_ENV === "development" ? "admin" : undefined);
if (!adminAccessCode) {
  return NextResponse.json({ valid: false, message: "Admin disabled" }, { status: 403 });
}
```

This preserves the convenient dev default while failing closed in production.

## Client-only read-only enforcement

The UI hides "Tambah Zona", "Tambah Barang", edit, and delete buttons when `isAdmin` is true — covering all visible mutation paths. However the item form submit handler still contains `school_id: isAdmin ? undefined : selectedSchool?.id`, and the API routes (`POST /api/zones`, item creation) have no server-side check rejecting requests where `school_id` is null or where the caller is admin-typed. A motivated user could craft requests to create ownerless records.

If the app's exposure grows, adding middleware that rejects write requests when the session is admin-typed would close the gap.
