# Admin mode, zone isolation, and single-zone report display

This change adds a lightweight admin mode that bypasses per-school filtering to load all schools' data, scopes zones to individual schools via a new `school_id` column and query filters, and swaps the report breakdown section from "Per Zona" to "Per Jenis" when viewing a single-zone report. The admin entry is hard-coded with the credential "admin"/"admin", authenticated entirely client-side via a shortcut in the verify endpoint.

Watch for: The admin credential is a hardcoded plaintext string with no rate-limiting or environment-based configuration (confirmed). The `zoneSlugExists` check doesn't filter by `school_id`, so slug collision avoidance operates globally even though the DB unique constraint is now scoped per school (confirmed). Admin mode allows creating items and zones with `school_id: null`, producing orphaned records that no school-scoped user will ever see (confirmed).

## High-level view

Admin mode is implemented as a special "admin" school entry that short-circuits the verify endpoint and triggers `isAdmin` in the React context. When `isAdmin` is true, every API call omits the `school_id` parameter, causing Supabase queries to return all rows unfiltered. The admin can view everything, but can also create new items and zones without a `school_id`, producing ownerless records.

Zone isolation adds a nullable `school_id` FK on the `zones` table with a composite unique index on `(slug, COALESCE(school_id, sentinel_uuid))`. The `listInventory` and `createInventoryZone` functions filter/scope by `school_id` when provided. However, the slug-uniqueness check in `createUniqueZoneSlug` still queries across all schools, meaning School A's "ruang-musik" will cause School B's zone to get suffixed to "ruang-musik-2" even though the DB would allow the same slug for different `school_id` values.

The report change is a one-liner conditional: when `isSingleZone` is true (derived from the presence of a `zoneId` query param), the breakdown shows "Per Jenis" (by type) instead of "Per Zona".

## Admin authentication is a static plaintext check

The verify endpoint (`src/app/api/schools/verify/route.ts`) accepts `schoolId === "admin" && accessCode === "admin"` and returns a success response with no interaction with Supabase or any secrets store. The credential is the same across all deployments, is visible in source code, and cannot be rotated without a code change and redeploy. There's no rate-limiting on the verify endpoint, so brute-forcing would work on the first attempt anyway given the known credential.

The admin entry is appended to the school list unconditionally:

```typescript
return NextResponse.json({ schools: [...(data ?? HARDCODED_SCHOOLS), ADMIN_ENTRY] });
```

Anyone who sees the school selector knows admin exists and can attempt `admin`/`admin`.

## Zone slug uniqueness diverges from DB constraint

The migration creates a composite unique index:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS zones_slug_school_id_unique
  ON zones (slug, COALESCE(school_id, '00000000-0000-0000-0000-000000000000'));
```

This allows the same slug across different schools. But the application-level uniqueness check in `createUniqueZoneSlug` calls `zoneSlugExists`, which queries:

```typescript
supabase.from("zones").select("id").eq("slug", slug).maybeSingle();
```

This matches any zone with that slug regardless of `school_id`. If School A already has "ruang-musik", School B will get "ruang-musik-2" even though the DB permits both to have "ruang-musik" with different `school_id` values. The `school_id` should be passed into `createUniqueZoneSlug` and `zoneSlugExists` to scope the check.

Additionally, `maybeSingle()` will throw if multiple rows match (which becomes possible now that the old unique constraint on slug alone is dropped and two schools can share the same slug). It should use `.limit(1)` or handle the multi-row case.

## Admin write path creates unscoped records

When `isAdmin` is true, the client sends `school_id: undefined` for both item and zone creation:

```typescript
school_id: isAdmin ? undefined : selectedSchool?.id,
```

In the store layer, this becomes `school_id: input.school_id || null` â€” a null FK. Items and zones with `school_id = NULL` will not appear when any school user queries with `.eq("school_id", theirId)`. They become invisible orphans unless another admin session fetches without a school filter.

The "Tambah Zona" button is hidden for admin, which prevents accidental zone creation. But the add-item form and zone edit/delete operations remain available to admin, so the orphan-creation path is still reachable via items, and edit/delete can mutate school-owned data without confirmation of target school.

## Condition logs query scales poorly for admin

In `listInventory`, when no `schoolId` is provided (admin mode):

```typescript
if (schoolId && activeItemIds.size > 0) {
  logsQuery = logsQuery.in("item_id", [...activeItemIds]);
}
```

When `schoolId` is falsy, neither branch applies, so the logs query fetches all condition logs unfiltered, then filters in-memory via `.filter((log) => activeItemIds.has(log.item_id))`. The admin view transfers the entire `item_condition_logs` table on every page load, with no server-side narrowing.
