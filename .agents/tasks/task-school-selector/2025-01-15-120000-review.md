# Netflix-style school selector with access-code gating

This change introduces multi-tenancy to the inventaris-paud app via a "school selector" flow. Users land on `/select-school`, pick a kindergarten from a card grid, enter a short numeric access code, and are redirected to the main inventory view filtered to that school's data. The session is persisted in localStorage via a React context backed by `useSyncExternalStore`. Server-side, a `schools` table with FK on `items` provides the isolation boundary, and API routes gain `school_id` filtering.

**Watch for:** Data isolation is enforced only on the read/create path — update and delete operations accept any item ID without verifying it belongs to the selected school (confirmed). Access codes are 1–2 digit numeric strings compared in plaintext, meaning the entire keyspace is trivially enumerable (confirmed). The hardcoded fallback school IDs (`school-1` through `school-5`) won't match the UUIDs in Supabase, so switching from fallback to live DB will orphan any items created during fallback mode (likely).

## High-level view

The school context uses `useSyncExternalStore` with a module-level cache and synthetic `StorageEvent` dispatch to trigger re-renders. Same-tab `localStorage` writes from outside this module (DevTools, other libraries) will go undetected unless they also dispatch the synthetic event.

Data isolation is applied at the `listInventory` and `createInventoryItem` level via an optional `school_id` parameter. However, the PUT `/api/items/[id]` and DELETE `/api/items/[id]` routes perform no school ownership check, meaning any client can modify or delete items belonging to another school if they know the item UUID.

The access code scheme uses plaintext 1–2 digit codes with no rate limiting or lockout. Since there are at most 99 possible codes per school, brute-force is trivial from a script.

The migration seeds data with `ON CONFLICT (id) DO NOTHING`, but the INSERT does not supply `id` values (they are auto-generated), so the conflict clause can never fire. Re-running the migration inserts duplicate rows.

## Cross-school mutation gap

The gap is in `src/app/api/items/[id]/route.ts`. The PUT handler calls `updateInventoryItem(id, body)` and the DELETE handler calls `softDeleteInventoryItem(id)` — both operate solely on the item UUID with no ownership check. A user who inspects network traffic from one school session could capture an item ID, switch schools (or craft a request), and mutate that item. The fix is to pass `school_id` into both functions and add `.eq('school_id', schoolId)` to their Supabase queries, causing a "not found" result if the item doesn't belong to the caller's school.

The read and create paths correctly enforce isolation: `listInventory(schoolId?)` applies `.eq('school_id', schoolId)` when present, and `createInventoryItem` persists the `school_id` field from the request body.

## Access code security posture

The codes are `"69"`, `"59"`, `"01"`, `"15"`, `"02"` — all numeric, at most two digits. The `/api/schools/verify` endpoint performs a single comparison per request with no rate limiting, lockout, or delay. An attacker (or curious parent) can enumerate all 99 possible codes for a given school in under a second.

For a low-stakes kindergarten app this may be acceptable, but the design frames it as "access code" implying restriction. If the goal is merely to route users to the correct school (not restrict access), the code prompt should be framed as a "confirmation" rather than a security gate. If actual restriction is intended, longer codes (6+ chars) or per-IP rate limiting are needed.

## School context storage coupling

The `selectSchool` and `clearSchool` callbacks mutate the module-scoped `cachedSchool` directly and then dispatch a synthetic `StorageEvent` on the same window — necessary because native `storage` events only fire in *other* tabs. The subscribe function listens for these events, invalidates the cache, and triggers a re-render. The coupling: any code that modifies the `inventaris-paud-school` localStorage key outside this module (DevTools, another library) will not trigger a re-render in the same tab unless it also dispatches the synthetic event. Cross-tab changes are detected normally via the native event.

## Migration idempotency issue

```sql
INSERT INTO schools (name, access_code)
VALUES (...)
ON CONFLICT (id) DO NOTHING;
```

The `id` column uses `DEFAULT gen_random_uuid()` and the INSERT does not supply explicit `id` values. PostgreSQL generates a new UUID for each row on every execution, so the `ON CONFLICT (id)` clause can never trigger. Running the migration a second time inserts 5 duplicate rows. Fix: add a UNIQUE constraint on `(name)` or `(name, access_code)` and conflict on that column instead.

## Condition logs fetch inefficiency

In `listInventory`, all `item_condition_logs` are fetched globally:

```typescript
supabase
  .from("item_condition_logs")
  .select("id, item_id, new_condition_id, notes, checked_by, checked_at")
  .order("checked_at", { ascending: false }),
```

When `schoolId` is specified, this still fetches logs for *all* schools' items and discards most of them client-side. For 5 schools with modest datasets this is negligible, but it scales linearly with total log volume across all schools. A subquery filtering by the items already scoped to the school would bound the fetch.

## Hardcoded fallback divergence

The schools API and verify routes maintain a `HARDCODED_SCHOOLS` array with IDs `school-1` through `school-5`, returned when Supabase is not configured. Items created during fallback mode store `school_id: "school-1"` (a plain string). The schema defines `school_id uuid REFERENCES schools(id)` — so the FK constraint would reject non-UUID values at insert time if Supabase were active. Developers testing with fallback mode should know their data isn't portable to production. A comment in the fallback or a documented migration path would clarify this boundary.
